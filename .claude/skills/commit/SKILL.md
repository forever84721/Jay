## Git Commit 流程

依序執行以下步驟：

### 1. 檢查目前狀態

```bash
git status
git diff --staged
git diff
```

### 2. 分析變更

- 辨識變更類型：新功能、修正、重構、文件、設定等
- 理解改動範圍：影響哪些模組、元件或功能
- 評估是否需要分批提交（不同關注點的變更應分開）

### 3. 暫存變更

若使用者未預先暫存，根據變更內容建議 `git add` 指令，並詢問確認後執行。

### 4. 撰寫提交訊息

遵循以下格式：

**標題（必填）**

```
<類型>: <簡潔描述>
```

- 長度限制：72 字元以內
- 使用動詞開頭（新增、修正、更新、移除、重構等）
- 不加句號

**類型對應**

| 類型       | 說明                     |
| ---------- | ------------------------ |
| `feat`     | 新功能                   |
| `fix`      | 錯誤修正                 |
| `refactor` | 重構（不影響外部行為）   |
| `style`    | 格式調整（不影響邏輯）   |
| `docs`     | 文件更新                 |
| `test`     | 新增或修改測試           |
| `chore`    | 建置流程、依賴套件更新等 |
| `perf`     | 效能改善                 |

**內文（選填）**

- 與標題之間空一行
- 說明「為什麼」，而非「做了什麼」（程式碼已說明 what）
- 每行不超過 72 字元

### 5. 執行提交

若**只有標題**，直接用 `-m`：

```bash
git commit -m "<類型>: <簡潔描述>"
```

若**有內文**，標題與內文之間必須空一行：

```bash
git commit -m "<類型>: <簡潔描述>

<內文說明為什麼>"
```

不確定是否需要內文時，以標題能完整表達為準，不需要強制補充。

### 6. 注意事項

- 提交訊息中**不要加** `Co-Authored-By` 或任何作者標記

### 7. 確認結果

```bash
git log --oneline -3
```
